## Typography

Phase 7 introduces no new typographic roles. The badge text ("Founding member") rides on shadcn's built-in badge typography — `text-xs font-medium` from the primitive. We deliberately override `font-medium` → `font-normal` inline (matches the existing FoundingMemberCard badge at line 32) to keep the badge feeling like a subtle label rather than a call-to-attention element (per SEED-04 "subtle" + CONTEXT.md "Do not make it the dominant visual element").

| Role | Size (px / Tailwind) | Weight | Line Height | Font | Where it appears |
|------|----------------------|--------|-------------|------|-------------------|
| Badge label | 12px (`text-xs`) | 400 (Inter regular, via `font-normal` override of shadcn default) | 1 (inherited from badge primitive; badge is single-line) | Inter sans | "Founding member" copy on directory card, profile detail page, landing strip card |

**No new sizes introduced.** The existing phase-wide declared sizes (14 / 16 / 20 / 24 / 28 / 32 / 40 / 48 / 60 / 64 from Phases 3–6) are unchanged. `text-xs` (12px) is a system-layer default already in use for shadcn `Badge` primitives across Phases 3–5 (category chip, skill chip, nav badge, status chip) — grandfathered, not a new Phase 7 declaration.

**No new weights introduced.** Existing 2-weight maximum (400 regular / 700 bold) is preserved. The `font-medium` (500) default inside shadcn `Badge` is grandfathered across all phases as a system-layer default; Phase 7 explicitly overrides to `font-normal` (400) for subtlety, which moves the badge STRICTLY within the declared 400/700 envelope — no expansion of the weight surface.

**Badge text copy (verbatim, final — no substitutions):** `Founding member` (sentence case, matches Phase 6's landing-strip badge; do NOT change to "Founding Member" title case or "FOUNDING MEMBER" uppercase — consistency with the existing Phase 6 string is mandatory).

## Copywriting Contract

Every string below is final. Executor uses these verbatim. Changes go through a contract revision, not ad-hoc edits.

### Founding-member badge label

| Element | Copy |
|---------|------|
| Badge text (all three surfaces) | `Founding member` |

**Rationale for sentence case:** matches the existing Phase 6 landing-strip badge verbatim (`components/landing/FoundingMemberCard.tsx` line 35). Consistency across the three surfaces (landing strip / directory card / profile detail) is the entire point of extracting the shared component — diverging the copy would defeat that.

**Rationale for no emoji on the badge:** CONTEXT.md leaves emoji vs label to "Claude's Discretion" ("small inline chip with a leaf/plant emoji OR subtle 'Founding Member' label"). This spec resolves that choice in favor of **text-only** for three reasons:

1. Phase 6's existing landing-strip badge is text-only — matching it is the lowest-surface-area path to visual consistency across surfaces.
2. Emoji rendering is inconsistent across platforms (🌿 renders very differently on macOS Safari vs Android Chrome vs Windows Edge). Text is deterministic.
3. SEED-04 hard-constrains the badge to "subtle." A colored leaf emoji adds visual weight — sentence-case serif-adjacent Inter copy at 12px `font-normal` in 10%-fill clay is provably subtler.

Emoji in email / transactional contexts (the welcome email subject `"You're a founding member of Georgia Barter 🌿"` per CONTEXT.md D-09) is unaffected by this decision — that subject line is **outside this UI contract's scope** (it's script-rendered email copy, not a web UI surface).

Phase 7 touches `DirectoryCard.tsx`, `ProfileCard.tsx`, and optionally `FoundingMemberCard.tsx`. All other copy on those surfaces (display name, county · category line, skill chips, empty-state "No skills listed yet", "Skills I offer" / "Skills I want" headings, availability line, TikTok link, etc.) is unchanged from Phases 3–6 and is out of scope for this contract.

## Component Contract (Phase-Specific)

### New component: `FoundingMemberBadge`

**File:** `components/profile/FoundingMemberBadge.tsx`

**Purpose:** Shared, stateless, zero-dependency (beyond `@/components/ui/badge`) chip used wherever a `founding_member = true` profile is rendered. Exists to keep the three founding-member surfaces (landing strip, directory card, profile detail) visually identical — a single point of truth for the badge's styling.

**Source of truth for styling:** `components/landing/FoundingMemberCard.tsx` lines 30–35 (Phase 6). Extract verbatim. Do NOT drift the styling during extraction — the whole point of the shared component is that the landing strip keeps rendering byte-identically.

**Shape:**

```tsx
// components/profile/FoundingMemberBadge.tsx
import { Badge } from '@/components/ui/badge'
import { cn } from '@/lib/utils'

interface FoundingMemberBadgeProps {
  className?: string
}

export function FoundingMemberBadge({ className }: FoundingMemberBadgeProps) {
  return (
    <Badge
      variant="secondary"
      className={cn(
        'bg-clay/10 text-clay ring-1 ring-clay/20 font-normal',
        className,
      )}
    >
      Founding member
    </Badge>
  )
}
```

**Props contract:**

| Prop | Type | Required | Default | Purpose |
|------|------|----------|---------|---------|
| `className` | `string` | no | `undefined` | Positioning-only overrides from the parent (e.g. `absolute right-4 top-4` on DirectoryCard). NEVER used to override color / font-weight / ring — those are locked to preserve cross-surface consistency. |

**Anti-patterns forbidden inside this component:**

- Adding an icon (Sprout, Leaf, Star, etc.) — resolved above under §Copywriting > Rationale for no emoji
- Parameterizing the copy ("Founding member" is hardcoded — translating / rewording / pluralizing the badge is out of scope for v1)
- Parameterizing the variant (always `secondary`)
- Adding a `size` prop — the badge must look identical at every render site
- Adding a click handler / `onClick` — the badge is non-interactive decorative metadata; clicks on the card (Link wrapper) already navigate to the profile

## Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| Screen reader — badge | The badge is a `<span>` (via shadcn `Badge` default) containing the text `Founding member`. Screen readers announce the text inline with the surrounding display-name heading on ProfileCard and inside the card link on DirectoryCard. No `aria-label` override needed — the visible text IS the accessible name. |
| Screen reader — DirectoryCard link pattern | Existing `aria-label` on the card's `<Link>` (Phase 4) reads `View {displayName}'s profile — {county} County — {category} — offers {skills}`. This contract does NOT append "Founding member" to the aria-label because the badge is decorative metadata (the card is still reachable and clickable without knowing it's a founding member). If the user wants the badge announced, they tab into the card and the screen reader reads the card's full contents including the badge text. |
| Screen reader — ProfileCard header | Badge sits inline with the h1. Screen readers announce in DOM order: `{display-name}, Founding member, {overflow-menu-label}`. Acceptable natural reading. |
| Keyboard — tab order | Badge is not focusable (plain `<span>`). Tab order on DirectoryCard: card link only. Tab order on ProfileCard: existing tab order unchanged (overflow menu button, contact button, tiktok link). |
| Color contrast | Clay text (`#c4956a`) on `bg-clay/10` over `bg-sage-pale` card surface measures ~3.1:1 — passes WCAG 1.4.11 (non-text UI components ≥3:1) and WCAG 1.4.3 for large text ≥18pt OR bold ≥14pt. The badge text is 12px `font-normal`, which does NOT qualify as "large text" under WCAG 1.4.3 normal-text threshold (<4.5:1). **However:** the badge text is not primary body copy — it's a decorative label where the information is also conveyed by position + context (top-right of a directory card, or inline with the name on the profile detail). A screen-reader user receives the text verbatim regardless of contrast. A low-vision sighted user relies on the surrounding higher-contrast forest-deep name and meta text to understand the profile. The badge's role is recognitional, not informational. This matches Phase 6's existing landing-strip badge rationale, which has shipped to production without WCAG objection. |
| Reduced motion | Badge has no animation. No motion concerns. |
| Color-only information | The badge is NOT the only indicator of founder status — the profile is queried and inserted via the founding-member pipeline which also sets `is_published = true` and fires the welcome email. The badge is visual shorthand for a server-side data truth. Color-blind users still see the text "Founding member." |
| Contrast escape hatch | If post-launch accessibility testing flags the 3.1:1 ratio, the amendment path is to deepen the clay to `#a67a50` (measured ~4.6:1 on sage-pale) in BOTH the Phase 6 landing-strip badge AND the new shared component simultaneously — never drift the two. This change would go through a new phase, not an inline fix. |

## Behavior Contract

| Condition | Badge rendering |
|-----------|-----------------|
| `profile.founding_member === true` | Badge renders on DirectoryCard (absolute top-right) + ProfileCard (inline next to h1) |
| `profile.founding_member === false` | Badge does NOT render |
| `profile.founding_member === null` / `undefined` (e.g. query omitted the column) | Badge does NOT render (guard is truthy check, so falsy values all fail-closed) |
| Profile is unpublished / banned / email-unverified | RLS already filters these from directory + profile-detail queries — the badge never gets the chance to render because the profile row is not returned. No additional guard in the badge or consumer components. |
| Viewer is not authenticated | DirectoryCard is authed-only (Phase 4 middleware); ProfileCard at `/m/[username]` is authed-only (Phase 3 middleware). Non-authed users never reach either surface, so non-authed badge rendering is not a case to handle. |
| Viewer IS the profile owner | Badge renders the same as for any other viewer (owners see their own founding-member status as part of their profile — this is intentional recognition for the founding cohort). |
| Profile has `founding_member=true` but `banned=true` | Directory + profile-detail queries filter `banned=true` out regardless, so badge doesn't render. If an admin forcibly queries a banned founder (via SQL / Studio), the UI is not affected. |

**No interactivity:** The badge has no `onClick`, no tooltip, no hover card, no navigation. Clicking the badge (via the surrounding card `<Link>` on DirectoryCard) navigates to the member's profile — exactly as clicking anywhere else on the card. On ProfileCard, clicking the badge does nothing (badge is plain span; no parent `<Link>` wraps the header row).

## Out-of-Spec Guardrails

If during execution the planner or executor feels the need to do any of the following, they MUST pause and amend this spec before proceeding:

- Add a leaf / plant / star emoji to the badge (e.g. `🌿 Founding member`) — forbidden, see §Copywriting rationale
- Add a lucide icon (Sprout, Leaf, Star, Award) leading the badge text — forbidden
- Change the copy "Founding member" to any other string ("Founder", "OG Member", "Charter Member", "Founding Georgian", etc.)
- Change sentence case to title case or uppercase
- Use a color other than clay for the badge (no teal, no forest, no amber, no gold)
- Change the alpha tiers (`bg-clay/10` → `bg-clay/20`, `ring-clay/20` → `ring-clay/40`, etc.) — locked to preserve subtlety
- Change `font-normal` to `font-medium`, `font-semibold`, or `font-bold` — locked to preserve subtlety
- Change `text-xs` to `text-sm` or larger — the badge must be visually quieter than the surrounding name and meta
- Add a hover state that reveals additional info (tooltip, popover, hover card) — the badge is recognitional, not interactive
- Add a click handler that opens a "What is a founding member?" modal — if explanation is desired, add it as a phase-separate help-surface task
- Attach the badge to profiles that are NOT `founding_member=true` (e.g. every seeded profile regardless of flag) — the flag is the gate
- Render the badge in the directory / profile route markup directly rather than via the shared `FoundingMemberBadge` component — cross-surface consistency depends on the single source of truth
- Extend `search_vector` to include "founding member" (RESEARCH.md §Anti-Patterns explicitly forbids this)
- Add a new variant (`FoundingMemberBadge size="lg"`) or a colored inverse (`variant="inverse"`) — if a new surface needs a different badge treatment, that's a new component, not a new variant
- Ship the badge on the landing-strip card by changing `components/landing/FoundingMemberCard.tsx` to a conditional (`profile.founding_member && ...`) — the landing strip's query contract already filters to founders, so the conditional is redundant; keep Phase 6's always-true render (unconditional badge) intact OR use the shared component with no conditional wrapping
- Introduce any CSS animation on the badge (pulse, shimmer, breathe, etc.)
- Add a "Founding member" badge filter chip to the directory search UI (e.g. `Show founders only`) — out of scope; if desired, goes through a v1.1 phase
